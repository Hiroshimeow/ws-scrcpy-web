# v0.1.25-beta.3 ÔÇö Manual VM Smoke Test Checklist

**Goal:** Verify everything that's shipped from v0.1.24 stable through v0.1.25-beta.3 still works end-to-end on a clean multi-user Win11 VM.

**Prerequisites on the VM:**
- Win11 VM with three accounts: `Admin` (local administrator), `User1` (standard user), `User2` (standard user). Take a clean snapshot before installing anything in case you need to retry.
- Browser available (Edge/Chrome/Firefox; Edge is fine).
- Optional but useful: `regedit` open for the registry checks. `Task Manager ÔåÆ Startup apps` tab handy.

**How to read this:**
- Each test has a setup, an action, and an expected outcome.
- Strikethrough or check off as you go. If anything fails, note which test and what diverged.
- Tests are grouped by feature area; some sections require state from a prior section (noted as "depends on").

---

## A ÔÇö Install / Upgrade Path

### A.1 ÔÇö Fresh install (clean snapshot)

**Setup:** revert VM to clean snapshot. Log in as `Admin`.

**Action:** Download and install `Setup-Win-x64.msi` (or whatever the v0.1.25-beta.3 installer is named) for v0.1.25-beta.3 from GitHub Releases.

**Expected:**
- [ ] Installer completes without errors.
- [ ] Browser auto-opens to `http://localhost:<port>` (typically 8000).
- [ ] WelcomeModal appears.
- [ ] App version in Settings ÔåÆ About reads `0.1.25-beta.3`.

### A.2 ÔÇö Upgrade from v0.1.24 stable (the migration path)

**Setup:** revert VM to clean snapshot. Install v0.1.24 stable (NOT beta.3 yet). Click "yes, install service" on the WelcomeModal as `Admin`. Confirm UAC prompt fires and uninstall service flow has not yet been tested (we're testing it later as the migration target).

**Action:** Either sideload the v0.1.25-beta.3 MSI, or use the in-app updater (Settings ÔåÆ Updates ÔåÆ Apply update).

**Expected:**
- [ ] Update applies cleanly.
- [ ] Browser tab refreshes / app reachable post-update.
- [ ] Admin's tray icon survives the update (may briefly flicker; one tray after settling).
- [ ] In `regedit`: `HKLM\Software\Microsoft\Windows\CurrentVersion\Run\WsScrcpyWebTray` is now PRESENT (the migration self-heal fired).
- [ ] In `regedit`: Admin's `HKCU\Software\Microsoft\Windows\CurrentVersion\Run\WsScrcpyWebTray` is still present at this point ÔÇö it'll be cleaned up at next admin logon (next test).

### A.3 ÔÇö Migration completes after admin logout/login

**Setup:** continuing from A.2.

**Action:** Log Admin out (don't shut down ÔÇö preserve service running). Log Admin back in.

**Expected:**
- [ ] At logon, BOTH HKLM and HKCU Run-keys fire briefly. Within a second or two, the per-session mutex resolves and only one tray icon remains visible.
- [ ] In `regedit`: Admin's `HKCU\...\Run\WsScrcpyWebTray` is GONE (Path C cleanup ran).
- [ ] Subsequent Admin logout/login cycles show exactly one tray, no flicker.

## B ÔÇö Multi-user Tray Coverage (regression guard for beta.1+beta.2)

### B.1 ÔÇö Non-admin user gets a tray at logon

**Setup:** Continuing from A.3 (or fresh install + admin-installs-service path).

**Action:** Switch to `User1` via Fast User Switching.

**Expected:**
- [ ] Tray icon appears in User1's system tray within ~2 seconds of logon.
- [ ] Right-click tray ÔåÆ "Open" ÔÇö browser opens to `http://localhost:<port>` and the app loads.

### B.2 ÔÇö Concurrent users each get their own tray

**Setup:** Continuing from B.1. Switch to `User2` (Admin and User1 still logged in if FUS supports it).

**Expected:**
- [ ] User2 also gets a tray icon at their logon.
- [ ] Each user's "Open" tray click navigates to the same `http://localhost:<port>` (same backend).
- [ ] Switching between users (Admin Ôåö User1 Ôåö User2) ÔÇö each session shows exactly ONE tray (per-session mutex working).

### B.3 ÔÇö Manually launching a second tray.exe is rejected

**Setup:** As any logged-in user with the tray running.

**Action:** Open `cmd` ÔåÆ `start "" "C:\Program Files\WsScrcpyWeb\current\ws-scrcpy-web-tray.exe"` (or right-click the exe in Explorer ÔåÆ Run).

**Expected:**
- [ ] No second tray icon appears.
- [ ] The spawned process exits silently within ~100ms (mutex acquire returned `Ok(None)`).
- [ ] Original tray remains responsive.

## D ÔÇö Settings ÔåÆ Service mode UX (new admin-confirm modal)

### D.1 ÔÇö Install service: confirm-modal cancel path

**Setup:** Fresh local-mode install (not service). OR uninstall service and switch to local mode first. Open Settings modal. Find the "install service" button under the Service section.

**Action:** Click "install service".

**Expected:**
- [ ] AdminConfirmModal appears with title "Administrative Privileges Required".
- [ ] Body text mentions UAC prompt and "Continue?".
- [ ] Two buttons: "cancel" and "continue".

**Action:** Click "cancel".

**Expected:**
- [ ] Modal closes.
- [ ] No UAC prompt fires.
- [ ] No fetch is made (Settings remains in same state, no spinner).

### D.2 ÔÇö Install service: confirm-modal Esc path

**Action:** Click "install service" again. Press `Esc`.

**Expected:**
- [ ] Modal closes; same as cancel-button outcome.

### D.3 ÔÇö Install service: confirm-modal backdrop-click path

**Action:** Click "install service" again. Click anywhere on the dimmed backdrop outside the modal card.

**Expected:**
- [ ] Modal closes; same as cancel-button outcome.

### D.4 ÔÇö Install service: confirm-modal continue ÔåÆ UAC

**Action:** Click "install service". Click "continue".

**Expected:**
- [ ] Modal closes immediately.
- [ ] Within ~1-2 seconds, Windows UAC prompt fires (yellow "Do you want to allow this app to make changes" dialog).

**Action (if Admin):** Click Yes on UAC.

**Expected:**
- [ ] Service installs.
- [ ] App auto-redirects to the service-mode URL (different port if scope is system).
- [ ] WelcomeModal does NOT re-appear.

### D.5 ÔÇö Uninstall service: confirm-modal + 5s "still waiting" affordance

**Setup:** Service mode active (from D.4 or fresh service install). Open Settings as `Admin`. Find the "uninstall service" button.

**Action:** Click "uninstall service" ÔåÆ continue.

**Expected:**
- [ ] UAC prompt fires.

**Action:** Click Yes on UAC.

**Expected:**
- [ ] Button text changes to "uninstallingÔÇª".
- [ ] If the user-session handoff is fast (<5s), the button stays "uninstallingÔÇª" briefly then redirects to local-mode URL.
- [ ] If the handoff takes longer (>5s), button text swaps to "still waiting for user sessionÔÇª" ÔÇö user knows progress isn't frozen.

### D.6 ÔÇö Uninstall service: non-admin silent-fail fix verification

**Setup:** Service mode active. Switch to `User1` (non-admin) via Fast User Switching. Open Settings.

**Action:** Click "uninstall service" ÔåÆ continue. Click Yes on UAC password prompt with admin credentials, OR type wrong password / cancel UAC to test the failure path.

**Expected (UAC accepted):**
- [ ] Service uninstalls successfully.
- [ ] App returns to local-mode UI.

**Expected (UAC declined):**
- [ ] Backend returns 403 + reason='uac-declined'.
- [ ] Frontend shows clear error message: "Administrative privileges were declined. Try again and approve the prompt."
- [ ] Button is no longer stuck on "uninstallingÔÇª" or "still waitingÔÇª" ÔÇö it returns to its prior state.

### D.7 ÔÇö Forced-handoff-failure path (artificial test)

**Setup:** Service mode active. As `Admin`, kill ALL `ws-scrcpy-web-tray.exe` processes via Task Manager (admin can see/kill all sessions' processes). Verify no tray helper running in any session via `Get-Process ws-scrcpy-web-tray` in PowerShell.

**Action:** Click "uninstall service" ÔåÆ continue.

**Expected:**
- [ ] Button shows "uninstallingÔÇª".
- [ ] After ~5 seconds, button swaps to "still waiting for user sessionÔÇª".
- [ ] After ~30 seconds total, backend's discover() times out.
- [ ] Frontend shows error: "Couldn't reach the user session. Make sure ws-scrcpy-web is running for your user, then try again."
- [ ] Button is freed (not stuck).
- [ ] Service is STILL INSTALLED (no silent direct-uninstall happened ÔÇö the load-bearing v0.1.25-beta.3 fix).
